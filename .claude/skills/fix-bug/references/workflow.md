# fix-bug 詳細ワークフロー

## ファイル構成と役割

| ファイル | 役割 | このスキルで触るか |
|----------|------|------------------|
| `docs/fix-log.md` | 修正履歴の追記専用ログ | 追記のみ（全体を読まない） |
| `docs/temp/fix-bug.md` | 今回のバグ報告 | 読んで、最後に削除 |
| `docs/adr/` | 設計判断の記録 | 設計変更を伴う場合のみ追記 |

## fix-log.md の追記フォーマット

```
## YYYY-MM-DD | #N | （一言説明）
症状: （何が起きていたか）
根本原因: （なぜ起きたか）
修正: （何を変えたか・ファイル名）
テスト: passed / skipped（理由）
ADR: なし or docs/adr/XXX.md
---
```

- `cat >> docs/fix-log.md` で追記。ファイル全体を読まない。

## 判断ルール

- 設計書（docs/CORE_RULES.md, docs/SYSTEM_SPEC.md 等）に跳ねる変更 → 計画提示・承認後に実装
- それ以外の修正（ロジックミス・タイポ・数値ズレ等） → 承認なしで進める

## テストについて

- `tests/` にテストファイルが存在する場合のみ実行
- テストが失敗した場合は修正を繰り返す（通るまで）
- 同じ箇所を3回以上修正しても通らない場合は人間に報告して止まる
- テスト未実装の場合はスキップ（ログに「テスト未実装」と記録）

## エラーパターン

| 状況 | 対処 |
|------|------|
| `fix-bug.md` が存在しない | `docs/temp/*.md` を確認。なければ人間に報告 |
| 承認が曖昧 | 「承認」の明示を求める。推測で進めない |
| 修正が別のバグを引き起こした | 元の修正を差し戻し、原因を報告してから再計画 |
| テストが何度修正しても通らない | 変更を差し戻し、状況を報告して人間に判断を仰ぐ |
